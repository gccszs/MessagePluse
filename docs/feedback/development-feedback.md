## 💡 设计决策记录

### 2026-03-02 - 传统功能融合设计

**背景**: MessagePulse 2.0 面向 AI 时代，但必须完整保留传统消息推送系统的核心功能

**讨论方式**: 头脑风暴模式（6个核心问题）

#### 决策 1：传统业务系统接入
**选择**: A - API 优先
**理由**:
- 统一的 REST API 接入方式，降低维护成本
- AI 系统和传统系统使用相同接口
- 提供多语言 SDK（Java、.NET、PHP、Python）简化接入

#### 决策 2：Web 管理界面范围
**选择**: C → A/B（完整功能愿景，分阶段实现）
**理由**:
- Phase A（最小化）：发送记录查询、状态查看、基础统计、API Key 管理、Skills 监控
- Phase B（标准）：消息模板管理、用户订阅管理、渠道配置、实时监控大屏
- Phase C（完整，愿景）：用户偏好设置、定时任务管理、批量发送界面、账单配额管理

#### 决策 3：定时和批量任务
**选择**: A - 集成 XXL-Job
**理由**:
- 简历中已提及 XXL-Job，保持一致性
- 成熟的分布式任务调度框架
- 支持定时任务、批量任务、延迟任务
- 支持任务依赖和失败重试

#### 决策 4：用户管理边界
**选择**: A + C（最小化管理 + 外部系统集成）
**理由**:
- 小型企业：使用最小化用户管理（接收者信息、订阅状态、偏好设置）
- 大型企业：对接现有用户系统（API 实时查询用户信息）
- 两种方式互补，覆盖不同场景

#### 决策 5：Flink 实时统计定位
**选择**: A - 保持可选组件
**理由**:
- 保留简历中的技术亮点
- 作为可选的学习/演示组件
- Docker Compose 配置中默认禁用
- 主要用于实时 QPS 统计、消息成功率趋势、链路追踪数据聚合

#### 决策 6：消息发送历史和查询
**选择**: A→B（API 优先，Web 提供实用查询功能）
**理由**:
- 保持 API 优先的设计哲学
- Web 界面提供比基础查询更完整的功能
- 分层查询方案：基础查询（Web）、高级查询（API）、运营查询（独立服务）
- 冷热分离：热数据（7天 MySQL）、温数据（30天 ES）、冷数据（90天+ 对象存储）

#### 传统功能保留清单
| 功能 | 状态 | 说明 |
|------|------|------|
| 双缓冲布隆过滤器去重 | ✅ 已实现 | 5分钟窗口，0.01%误判率 |

## 🤝 协作指南

### 给其他 Agent 的建议

1. **开始工作前**:
   - 阅读完整设计文档
   - 检查本反馈文档的最新更新
   - 确认当前 Phase 和任务分配

2. **遇到问题时**:
   - 按照反馈格式模板记录问题
   - 尝试至少 2 种解决方案
   - 记录最终方案和经验教训

3. **提交代码前**:
   - 更新反馈文档
   - 标注相关代码位置
   - 更新进度跟踪表

4. **沟通建议**:
   - 重大决策记录在"设计决策记录"部分
   - 紧急问题标记为 P0 优先级
   - 保持文档同步更新
