# 消息撤回功能设计方案

## 问题分析

"消息撤回"在消息推送领域存在技术限制：

| 发送阶段 | 可撤回性 | 原因 |
|----------|----------|------|
| 未进入队列 | ✅ 完全可撤回 | 消息尚未处理 |
| Kafka队列中 | ✅ 可撤回 | 消息未被消费 |
| 消费者处理中 | ⚠️ 部分可撤回 | 需要检查是否已调用渠道 |
| 已调用第三方 | ❌ 不可撤回 | 第三方渠道不可控 |
| 已送达用户 | ❌ 不可撤回 | 短信/邮件已发出 |

## 设计方案：分阶段撤回机制

### 整体架构

```
用户请求撤回
    ↓
查询消息状态
    ↓
┌───┴───┬─────────┬──────────┐
│       │         │          │
待发送  处理中    已发送    已送达
│       │         │          │
│       ↓         ↓          ↓
│   取消费用  发送撤销通知   提示不可撤回
│       │      (如邮件撤回请求)
│       ↓         ↓
│   标记已撤回   记录撤回尝试
│
从Kafka删除或标记为撤回
```

### 具体实现

#### 1. 消息状态管理

```java
public enum MessageStatus {
    PENDING,        // 待发送（在Kafka中）
    PROCESSING,     // 处理中（消费者已取出）
    SENT_TO_CHANNEL, // 已发送到第三方渠道
    DELIVERED,      // 已送达用户
    FAILED,         // 发送失败
    REVOKED         // 已撤回
}

@Entity
public class Message {
    private String id;
    private MessageStatus status;
    private String channelId;  // 渠道消息ID（如有）
    private LocalDateTime createTime;
    private LocalDateTime sendTime;  // 实际发送时间
    // ... 其他字段
}
```

#### 2. 撤回接口设计

```java
@Service
public class MessageRevokeService {

    @Autowired
    private KafkaTemplate<String, Message> kafkaTemplate;

    @Autowired
    private MessageRepository messageRepository;

    /**
     * 撤回消息
     * @param messageId 消息ID
     * @return 撤回结果
     */
    public RevokeResult revokeMessage(String messageId) {
        Message message = messageRepository.findById(messageId)
            .orElseThrow(() -> new MessageNotFoundException(messageId));

        // 根据状态判断是否可撤回
        switch (message.getStatus()) {
            case PENDING:
                // 从Kafka删除或标记撤回
                return revokeFromKafka(message);

            case PROCESSING:
                // 检查是否已调用渠道
                if (message.getChannelId() == null) {
                    return revokeFromConsumer(message);
                } else {
                    return tryChannelRevoke(message);
                }

            case SENT_TO_CHANNEL:
                return tryChannelRevoke(message);

            case DELIVERED:
            case FAILED:
                return RevokeResult.notSupported("消息已送达或失败，无法撤回");

            case REVOKED:
                return RevokeResult.alreadyRevoked();

            default:
                return RevokeResult.unknown();
        }
    }

    // 从Kafka撤回
    private RevokeResult revokeFromKafka(Message message) {
        // 方案1：标记为撤回，消费者检查标记后跳过
        message.setStatus(MessageStatus.REVOKED);
        messageRepository.save(message);

        // 方案2：发送撤回命令到Kafka（优先级更高）
        RevokeCommand command = new RevokeCommand(message.getId());
        kafkaTemplate.send("revoke-topic", command);

        return RevokeResult.success("消息已从队列撤回");
    }

    // 从消费者撤回（未调用渠道）
    private RevokeResult revokeFromConsumer(Message message) {
        // 通过Redis发布撤回事件
        redisTemplate.convertAndSend(
            "message:revoke:" + message.getId(),
            "REVOKE"
        );
        message.setStatus(MessageStatus.REVOKED);
        messageRepository.save(message);
        return RevokeResult.success("消息处理已取消");
    }

    // 尝试通过渠道撤回（有限支持）
    private RevokeResult tryChannelRevoke(Message message) {
        // 邮件：发送撤回请求（Gmail支持）
        if (message.getChannelType() == ChannelType.EMAIL) {
            channelRevokeService.sendRevokeRequest(message);
            return RevokeResult.partial(
                "已发送撤回请求，但不保证成功"
            );
        }

        // 短信/微信：不支持撤回
        return RevokeResult.notSupported(
            "已发送到第三方渠道，无法撤回"
        );
    }
}
```

#### 3. 消费者端的撤回检查

```java
@KafkaListener(topics = "message-topic")
public void handleMessage(Message message) {
    // 发送前检查是否已撤回
    if (messageRepository.isRevoked(message.getId())) {
        log.info("Message {} is revoked, skip sending", message.getId());
        return;
    }

    // 订阅撤回事件
    String revokeKey = "message:revoke:" + message.getId();
    redisTemplate.subscribe(new MessageListener() {
        @Override
        public void onMessage(String message, String pattern) {
            if ("REVOKE".equals(message)) {
                // 取消发送
                throw new RevokeException("Message revoked");
            }
        }
    }, revokeKey);

    try {
        // 更新状态为处理中
        message.setStatus(MessageStatus.PROCESSING);
        messageRepository.save(message);

        // 调用渠道API
        SendResult result = channelAdapter.send(message);

        // 保存渠道消息ID
        message.setChannelId(result.getChannelMessageId());
        message.setStatus(MessageStatus.SENT_TO_CHANNEL);
        messageRepository.save(message);

    } catch (RevokeException e) {
        message.setStatus(MessageStatus.REVOKED);
        messageRepository.save(message);
    }
}
```

#### 4. 前端展示

```javascript
// 消息详情页面
function showMessageDetail(messageId) {
  const message = await api.getMessage(messageId);

  // 根据状态显示撤回按钮
  if (['PENDING', 'PROCESSING'].includes(message.status)) {
    showRevokeButton();  // 显示撤回按钮
  } else if (message.status === 'SENT_TO_CHANNEL') {
    showPartialRevokeButton();  // 显示部分撤回按钮（邮件）
  } else {
    hideRevokeButton();  // 隐藏撤回按钮
  }
}

// 撤回操作
async function revokeMessage(messageId) {
  const result = await api.revokeMessage(messageId);

  if (result.success) {
    showSuccess('消息已撤回');
  } else if (result.partial) {
    showWarning(result.message);  // 已发送撤回请求，但不保证成功
  } else {
    showError(result.message);   // 无法撤回
  }
}
```

### 方案特点

| 特性 | 说明 |
|------|------|
| **分阶段处理** | 根据消息状态采用不同撤回策略 |
| **最大兼容性** | 利用邮件渠道的撤回能力 |
| **用户体验** | 明确告知哪些情况可撤回 |
| **实现简单** | 不需要修改核心发送流程 |

### 在面试中的表述

"关于消息撤回，我设计了分阶段撤回机制。消息在Kafka队列中可以完全撤回，消费者处理中但未调用渠道时可以通过Redis事件取消，已发送到渠道的则视渠道能力而定。比如邮件可以发送撤回请求，但短信和微信就不支持。我在前端会根据消息状态动态显示撤回按钮，给用户明确的预期。"

## 替代方案：补偿消息

如果无法实现真正的撤回，可以考虑"补偿消息"方案：

```java
// 发送撤回通知
public void sendRevokeNotification(String originalMessageId) {
    Message notification = Message.builder()
        .type(MessageType.REVOKE_NOTIFICATION)
        .content("之前的消息已撤回")
        .originalMessageId(originalMessageId)
        .build();

    kafkaTemplate.send("message-topic", notification);
}
```

这种方式适用于：
- 营销类消息（撤回通知替代原消息）
- 非紧急通知（告知用户原消息作废）
- 对时效性要求不高的场景
