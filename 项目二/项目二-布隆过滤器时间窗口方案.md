# 布隆过滤器时间窗口处理方案

## 问题分析

布隆过滤器本身不支持删除操作，如果用于基于时间窗口的去重（如5分钟内不能重复发送），会面临以下问题：
- 过期消息ID不断累积，导致误判率上升
- 过滤器最终饱和，失去去重能力

## 推荐方案：滑动窗口 + 双缓冲布隆过滤器

### 核心思路

使用两个布隆过滤器交替工作，定期切换和清理：

```
时间轴：  0 min    5 min    10 min   15 min
         |--------|--------|--------|
当前窗口：   BloomFilter-A (活跃)
历史窗口：            BloomFilter-B (待清理)
切换时刻：    ↑        ↑        ↑
         重建B     重建A     重建B
```

### 具体实现

```java
public class TimeWindowBloomFilter {
    private BloomFilter<String> currentFilter;
    private BloomFilter<String> previousFilter;
    private volatile long windowStartTime;
    private final long windowDurationMs = 5 * 60 * 1000; // 5分钟窗口

    // 初始化
    public TimeWindowBloomFilter() {
        // 预估元素数量100万，误判率0.01%
        currentFilter = BloomFilter.create(Funnels.stringFunnel(Charset.defaultCharset()),
                                         1_000_000, 0.0001);
        previousFilter = BloomFilter.create(Funnels.stringFunnel(Charset.defaultCharset()),
                                          1_000_000, 0.0001);
        windowStartTime = System.currentTimeMillis();
    }

    // 检查是否重复
    public boolean mightContain(String messageId) {
        // 先查当前窗口
        if (currentFilter.mightContain(messageId)) {
            return true;
        }
        // 再查历史窗口（防止切换时的边界问题）
        if (previousFilter.mightContain(messageId)) {
            return true;
        }
        return false;
    }

    // 添加消息ID
    public void put(String messageId) {
        currentFilter.put(messageId);
        checkAndRotateWindow();
    }

    // 窗口切换逻辑
    private void checkAndRotateWindow() {
        long now = System.currentTimeMillis();
        if (now - windowStartTime >= windowDurationMs) {
            synchronized (this) {
                // 双重检查
                if (now - windowStartTime >= windowDurationMs) {
                    // 切换窗口：当前变历史，重建新的当前
                    previousFilter = currentFilter;
                    currentFilter = BloomFilter.create(
                        Funnels.stringFunnel(Charset.defaultCharset()),
                        1_000_000, 0.0001
                    );
                    windowStartTime = now;
                }
            }
        }
    }
}
```

### 方案优势

| 特性 | 优势 |
|------|------|
| **内存可控** | 每个窗口只有100万个元素，约1.2MB内存 |
| **误判率稳定** | 定期重建，误判率始终保持在设计值 |
| **无阻塞** | 切换过程快速，不影响正常请求 |
| **边界安全** | 双缓冲防止切换瞬间的漏判 |

### 参数选择建议

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| 窗口时长 | 5分钟 | 覆盖大部分重复场景 |
| 预估元素数 | 100万 | 根据5分钟内峰值QPS估算 |
| 误判率 | 0.01% | 平衡内存和准确性 |

### 性能分析

```
内存占用：
- 每个布隆过滤器：约1.2MB
- 双缓冲总计：约2.4MB
- 相比Redis方案节省网络开销

CPU开销：
- mightContain: O(1) 哈希计算
- put: O(1) 哈希计算 + 偶尔的窗口切换

切换开销：
- 每5分钟一次
- 重建耗时约10-50ms
- 对99.99%的请求无影响
```

## 备选方案对比

| 方案 | 优势 | 劣势 | 推荐指数 |
|------|------|------|----------|
| 双缓冲布隆过滤器 | 内存小、性能好、实现简单 | 需要预估容量 | ⭐⭐⭐⭐⭐ |
| Counting Bloom Filter | 支持删除 | 内存占用大（4-5倍） | ⭐⭐⭐ |
| Redis过期Key | 原生支持TTL | 网络开销、成本高 | ⭐⭐⭐⭐ |
| 定时全量重建 | 实现最简单 | 重建期间性能抖动 | ⭐⭐ |

## 在面试中的表述

"对于布隆过滤器的时间窗口问题，我采用了双缓冲方案：使用两个布隆过滤器交替工作，每5分钟切换一次。当前窗口记录最近5分钟的消息ID，历史窗口用于防止切换瞬间的边界问题。这样既能保证去重效果，又能控制内存占用和误判率。"
