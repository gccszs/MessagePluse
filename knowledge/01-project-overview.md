# 项目概述

## 项目定位

MessagePulse 2.0 是面向 AI 时代的统一消息分发平台，核心价值是**解耦 AI 系统与消息渠道**。

### 核心使命

- **AI 系统侧**：OpenClaw、Cursor、Claude Code 等 AI 框架只需开发一个 MessagePulse 适配器，即可接入所有下游渠道
- **消息渠道侧**：短信、邮件、微信、飞书、钉钉等渠道以独立 Skill 形式运行，按需激活和部署
- **平台核心**：提供统一的消息去重、链路追踪、分阶段撤回、死信队列等基础能力

## 设计哲学

### 传统模式的问题

```
传统模式（复杂）：
OpenClaw → 短信插件（复杂）
         → 邮件插件（复杂）
         → QQ插件（复杂）
         → 微信插件（复杂）
```

每个 AI 系统需要为每个渠道开发独立插件，维护成本高，扩展困难。

### MessagePulse 2.0 解决方案

```
新方案（解耦）：
OpenClaw → MessagePulse 插件（统一接口）
         → MessagePulse → 短信/邮件/QQ/微信/飞书
```

AI 系统只需对接 MessagePulse 一个平台，所有渠道由平台统一管理。

### 核心优势

- 🔌 **插件化**：每个渠道独立 Skill，按需激活
- 🔀 **解耦**：AI 系统无需关心具体渠道
- 📦 **可扩展**：新增渠道 = 新增 Skill
- 🎛️ **用户可控**：想接入飞书？激活飞书 Skill

## 目标用户

### 主要用户群体

1. **AI 系统开发者**
   - OpenClaw、Cursor、Claude Code 等 AI 框架开发者
   - 需要为 AI 系统添加消息通知能力
   - 希望快速接入多种消息渠道

2. **企业开发团队**
   - 需要统一的消息推送平台
   - 管理多种消息渠道（短信、邮件、企业IM）
   - 需要可靠的消息投递保障

3. **SaaS 服务提供商**
   - 为客户提供消息通知服务
   - 需要多租户隔离
   - 需要计费和配额管理

## 核心价值

### 1. 降低接入成本

- AI 系统只需开发一个 MessagePulse 适配器
- 无需为每个渠道单独开发插件
- 统一的 API 接口，学习成本低

### 2. 提升可维护性

- 渠道逻辑独立封装在 Skill 中
- 渠道升级不影响 AI 系统
- 集中式配置管理

### 3. 增强可扩展性

- 新增渠道无需修改核心代码
- 支持动态激活/停用渠道
- 支持自定义 Skill 开发

### 4. 保障消息可靠性

- 三级去重机制（布隆过滤器 + Redis + MySQL）
- 消息状态机管理
- 自动重试和死信队列
- 完整的链路追踪

### 5. 支持传统系统

- 向后兼容传统消息推送功能
- 提供多语言 SDK（Java、.NET、PHP、Python）
- 支持定时任务、批量发送
- Web 管理界面

## 与其他系统的关系

### 依赖的外部系统

- **Kafka**：消息队列，核心数据流转
- **MySQL**：持久化存储，消息状态管理
- **Redis**：缓存、去重、限流
- **渠道服务商**：短信、邮件、企业IM 等第三方服务

### 被依赖的系统

- **AI 框架**：OpenClaw、Cursor、Claude Code
- **业务系统**：传统 Java、.NET、PHP 应用
- **监控系统**：Prometheus、Grafana、SkyWalking

### 可选集成

- **XXL-Job**：定时任务调度
- **Flink**：实时统计分析
- **Elasticsearch**：消息查询和全文搜索
- **外部用户系统**：企业现有用户管理系统
